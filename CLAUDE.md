# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A Power BI dashboard (PBIP mode) for the hospitality industry — the "DEEPLY Hospitality Executive Dashboard" (see [Deeply_Full_Specification.pdf](Deeply_Full_Specification.pdf)) built for a fictional hotel "The Grand Azure" (Mid-Scale, 200 rooms, Tel Aviv, USD). Phase 1 covers four pages: Executive, Commercial, Finance, Profitability. Hardcoded demo data for 2025–2026 is generated by [generate_model.py](generate_model.py) and lives inline in the TMDL M partitions — there is no external data source. UI is English only.

## pbi-cli — always use it for Power BI work

`pbi-cli` is installed globally and is the primary tool for editing this project:

- Binary: `C:\Users\marce\.local\bin\pbi.exe` (also aliased as `pbi-cli.exe`)
- Version: `pbi --version` (currently 3.10.5)
- Top-level commands: `dax`, `measure`, `relationship`, `table`, `column`, `partition`, `calendar`, `calc-group`, `hierarchy`, `expression`, `perspective`, `security-role`, `report`, `bookmarks`, `filters`, `format`, `model`, `database`, `connect`, `setup`

Use `pbi` **before** hand-editing TMDL / PBIR JSON files — it validates schemas, handles lineage tags, and manages transactions. Specialized skills are registered in Claude Code and match each command group: `power-bi-dax`, `power-bi-modeling`, `power-bi-deployment`, `power-bi-docs`, `power-bi-partitions`, `power-bi-security`, `power-bi-diagnostics`, `power-bi-report`, `power-bi-visuals`, `power-bi-pages`, `power-bi-themes`, `power-bi-filters`.

**Semantic model commands** (`measure`, `relationship`, `table`, `column`, `dax`, etc.) require a live connection to Power BI Desktop:
```bash
pbi connect    # auto-detects running PBI Desktop instance
```

**Report commands** (`report`, `bookmarks`, `filters`, `format`) operate directly on the PBIR JSON files — no connection needed.

## Star schema model

5 fact tables, 8 dimensions, 1 hidden `_Measures` table with 44 DAX measures organized into display folders (Revenue, Occupancy, Profitability, Budget, Time Intelligence, Bookings, Channels, Guest).

Key fact-specific gotchas:
- `Fact_Revenue` has **no SegmentID or ChannelID** — breakdowns by segment/channel must use `Fact_Reservations` via `Reservation Gross Revenue` / `Reservation Net Revenue` measures.
- `Fact_Occupancy` is the source for Occupancy %, ADR, RevPAR, GOPPAR — a dedicated monthly table (24 rows) because `Fact_Reservations` alone was too sparse.
- Only `Fact_Reservations.ArrivalDateID → Dim_Date` is related (not DepartureDateID, BookingDateID, CancellationDateID) to avoid ambiguous paths.

## Critical TMDL rules learned the hard way

- **Never define relationships in both `model.tmdl` and `relationships.tmdl`.** The PBI engine treats them as duplicates and throws `"ambiguous paths: X→Y and X→Y"` (same path shown twice). Relationships belong **only** in `relationships.tmdl`.
- **Multi-line DAX (VAR/RETURN)** must be indented at **3 tabs** — one deeper than measure properties (which are at 2 tabs). Otherwise the parser reports `"Token ',' expected"` or similar M-engine errors. Cannot be passed via `-e` — use `--file` or stdin piping.
- **Inline M expressions** must use `source =` (with the equals sign). Plain `source` followed by `let` breaks with `"keyword 'let' is neither a property nor an object"`.
- **`#table(...)` for empty record types** uses `{}`, not `[]` — e.g. `#table({"_dummy"}, {})` for the hidden `_Measures` table.
- **Sort a month/day column** with `sortByColumn: Month` on the MonthName column. Without this, charts show Apr/Aug/Dec alphabetical order.

## Critical PBIR rules learned the hard way

- **`themeCollection.customTheme` requires `reportVersionAtImport`** — same three fields (`visual`, `report`, `page`) as `baseTheme`. Without it, publish fails with `"Required properties are missing from object"`.
- **Don't put `layoutOptimization`** at the report level — not in the 3.2.0 schema.
- **Don't include `Commands`** inside `visual.query` — not in the 2.7.0 visual schema.
- **Slicer column bindings** use `"Column"` field type (not `"Measure"`) — a `Dim_Date[Year]` bound as Measure shows wrong values.
- **Scatter chart split points** use roles `Category` + `Series`, not `Details`, to produce one point per group.

## Data refresh is required after model changes

Adding measures, relationships, or columns defines them but doesn't compute them. After publishing a change:
1. Open `Report.pbip` in Power BI Desktop → **Home → Refresh** before saving.
2. Or refresh the dataset in the Power BI Service.

Symptom when skipped: visuals show `(Blank)` or `"Error fetching data for this visual — relationship ... does not hold any data because it needs to be recalculated"`.

## Common tasks

### Regenerate the model from scratch
```bash
python generate_model.py
```
Rewrites all 13 table files in `Report.SemanticModel/definition/tables/` (uses `random.seed(42)`). Does not touch `relationships.tmdl`, `model.tmdl`, or `_Measures.tmdl` hand-edits — but will overwrite table structure.

### Preview the published report
The output is published to Power BI Service. Use playwright MCP tools to inspect UX (navigate, screenshot, snapshot). The `view?r=...` links are public sharing URLs.
